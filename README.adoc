= Enhancing microservices with custom tracing and metrics using OpenTelemetry and Grafana

[.hidden]
NOTE: This repository contains the guide documentation source. To view the guide in published form, view it on the https://openliberty.io/guides/{projectid}.html[Open Liberty website].

Learn how to add manual instrumentation to collect custom spans in traces and custom metrics from microservices using MicroProfile Telemetry and the Grafana stack.

// =================================================================================================
//  What you'll learn
// =================================================================================================
== What you'll learn

Automatic instrumentation in MicroProfile Telemetry makes it easy to capture traces, metrics, and logs without changing your code. However, it might not capture all the details you need. In many cases, you might want to monitor specific operations, track custom events, or measure performance indicators unique to your application.

Manual instrumentation lets you add custom spans and metrics directly to your code. These custom signals are collected alongside the automatically captured data, allowing you to view and analyze everything together in the same observability backend.

In this guide, you'll learn how to add manual instrumentation to your microservices by creating ccustom spans in traces and custom metrics to extend the telemetry collected automatically. The guide uses the https://github.com/grafana/docker-otel-lgtm/?tab=readme-ov-file#docker-otel-lgtm[Grafana Docker OpenTelemetry LGTM^] image, an open source Docker image that provides a preconfigured observability backend for OpenTelemetry, based on the https://grafana.com/about/grafana-stack/[Grafana stack^].

This guide builds on the https://openliberty.io/guides/microprofile-telemetry-grafana-automatic.html[Enabling observability in microservices with traces, metrics, and logs using OpenTelemetry and Grafana^] guide. If you are not familiar with enabling automatic telemetry collection, it will be helpful to read that guide before you proceed.

The diagram shows multiple services, but for simplicity, this guide configures only the `system` and `inventory` services to demonstrate observability in a distributed environment.

== Additional prerequisites 

Before you begin, ensure that Docker is installed and running on your system. For installation instructions, see the https://docs.docker.com/get-started/get-docker/[official Docker documentation^].

Start a container from the https://hub.docker.com/r/grafana/otel-lgtm[grafana/otel-lgtm^] Docker image by running the following command:

[role="command"]
----
docker run -p 3000:3000 -p 4317:4317 -p 4318:4318 --rm -ti grafana/otel-lgtm
----

It may take a minute for the container to start. After you see the following message, your observability stack is ready:

[role="no_copy"]
----
The OpenTelemetry collector and the Grafana LGTM stack are up and running.
----

Once the container is running, you can access the Grafana dashboard at the http://localhost:3000 URL.

The [hotspot=listMethod file=1]`list()`, [hotspot=addMethod file=1]`add()`, and [hotspot=updateMethod file=1]`update()` methods are annotated with the [hotspot=addWithSpan hotspot=listWithSpan hotspot=updateWithSpan file=1]`@WithSpan` annotation. === Creating custom spans

The MicroProfile Telemetry specification makes the underlying OpenTelemetry Tracer instance available. You can create spans manually by injecting the OpenTelemetry Tracer into your application.

----
#Replace the `InventoryResource` class:#
`inventory/src/main/java/io/openliberty/guides/inventory/InventoryResource.java`
----

InventoryResource.java
[source, java, linenums, role='code_column hide_tags=copyright']
----
include::finish/inventory/src/main/java/io/openliberty/guides/inventory/InventoryResource.java[]
----

Inject the OpenTelemetry [hotspot=tracer file=0]`Tracer` interface into the [hotspot file=0]`InventoryResource` class to manually instrument your code to collect traces. Before the [hotspot=manager hotspot=getHealth file=0]`InventoryManager` calls the `system` service to check its health status, it creates and starts a span named [hotspot=getPropertiesSpan file=0]`GettingProperties`.

Each span must be completed by calling [hotspot=end file=0]`end()`. To ensure the span is always ended, the call is placed inside a [hotspot=finally file=0]`finally` block.

The [hotspot=scope file=0]`makeCurrent()` method sets the span as the current span. This call returns a `Scope` that must be closed to restore the previous span. A [hotspot=try file=0]`try-with-resources` block is used to close the scope automatically.  

Use the [hotspot=addEvent1 hotspot=addEvent2 file=0]`addEvent()` Span API to add events to the span. In this example, the code adds one event when the properties are received and another when the request fails.

For more information about OpenTelemetry distributed tracing, see the https://www.javadoc.io/doc/io.opentelemetry/opentelemetry-api/latest/io/opentelemetry/api/trace/package-summary.html[OpenTelemetry distributed tracing API documentation^]. You can also refer to the https://openliberty.io/guides/microprofile-telemetry-jaeger.html[Enabling distributed tracing in microservices with OpenTelemetry and Jaeger^] guide for a more detailed example of tracing instrumentation in Open Liberty.

To see these spans and events, visit the http://localhost:9081/inventory/systems/localhost URL and open the Grafana dashboard at the http://localhost:3000 URL. In the **Explore** view, select the **Tempo** data source and **Run query**. Click a trace ID with the `GET /inventory/systems/{hostname}` name that you just generated. You’ll see the `GettingProperties` span.

image::inventory_getting_properties_span.png[Inventory Manager GettingProperties span,align="center",align="center"]
{empty} +

To test a failure case, go to the http://localhost:9081/inventory/systems/unknown URL and run the same trace query in Grafana. Click a trace ID with the `GET /inventory/systems/{hostname}` name that you just generated. There are two spans from the `inventory` service. You'll see the `GettingProperties` span. Expand the `GettingProperties` span and its **Events** field. You can see the `Cannot get properties` event.

image::inventory_cannot_get_properties_event.png[Inventory Manager Cannot get properties event,align="center"]
{empty} +

Inject the OpenTelemetry [hotspot=meter file=1]`Meter` interface into the [hotspot file=1]`HealthCheckScheduler` class to define and collect application-specific metrics. This interface provides access to the OpenTelemetry metrics API, including instruments such as counters, histograms, and gauges to monitor application behavior and performance.

Use the [hotspot=counterBuilder file=1]`Meter.counterBuilder()` method to define a counter instrument named [hotspot=schedulerRunCounter file=1]`inventory.health_check.scheduler.runs`. This metric tracks a cumulative number of the health check scheduler executions.

Use the [hotspot=histogramBuilder file=1]`Meter.histogramBuilder()` method to define a histogram instrument named [hotspot=schedulerDuration file=1]`inventory.health.check.scheduler.duration`. This metric measures the duration of scheduled health check runs. The [hotspot=start file=1]`start` time is recorded when the [hotspot=performHealthChecks file=1]`performHealthChecks()` method begins, and the [hotspot=duration file=1]`duration` is calculated and recorded when the health check perform finishes.

Inject the OpenTelemetry [hotspot=meter file=2]`Meter` interface and use the [hotspot=gaugeBuilder file=2]`Meter.gaugeBuilder()` method to define a gauge named [hotspot=inventorySize file=2]`inventory.size`, recording the number of systems currently in the inventory. The [hotspot=buildWithCallback file=2]`buildWithCallback()` method allows the value to be updated dynamically, ensuring that the metric always reflects the current size of the list.

When naming metrics, follow OpenTelemetry's https://opentelemetry.io/docs/specs/semconv/general/metrics/[general naming guidelines] to ensure compatibility. 

To learn more about how to define and use metrics in your application, refer to the https://opentelemetry.io/docs/languages/java/api/#meter[Meter operations^] for a full list of available metrics, or the https://www.javadoc.io/doc/io.opentelemetry/opentelemetry-api/latest/io/opentelemetry/api/metrics/package-summary.html[OpenTelemetry metrics API reference^].

Now, to view the custom metrics, first trigger some activity in the application by visiting the following URLs in your browser:

* Visit the http://localhost:9081/inventory/systems/localhost URL to add your local system properties to the inventory.

* Visit the http://localhost:9081/inventory/systems/127.0.0.1 URL to add the system properties for host `127.0.0.1` to the inventory.

* Visit the http://localhost:9080/system/unhealthy URL to set localhost and also host `127.0.0.1` health status to `DOWN`.

Next, open the Grafana dashboard at the http://localhost:3000 URL. From the left menu, navigate to **Drilldown -> Metrics**, then enter `inventory` in the **Quich search metrics** field to explore the metrics exposed by the service. In the upper right corner of the Grafana dashboard, click **Set auto refresh interval** and choose **30s** for real-time updates.

Note that OpenTelemetry metrics are sent every 60 seconds by default. After triggering activity, it may take up to a minute for your custom metrics to appear in Grafana. If you don't see the metrics immediately, wait for a short period for the dashboard to refresh.

You will see a result similar to the following:

image::inventory_custom_metrics.png[Inventory custom metrics,align="center"]
{empty} +

Note the difference between the metric name you define and the name displayed in Grafana. For details on how OpenTelemetry metrics are transformed when exported to Prometheus, see the https://opentelemetry.io/docs/specs/otel/compatibility/prometheus_and_openmetrics/#otlp-metric-points-to-prometheus[OTLP Metric points to Prometheus^] section of the OpenTelemetry specification.

The health check scheduler is scheduled to run every 30 seconds. Wait for a few minutes and refresh the Grafana dashboard to see how the metrics update in real time.
